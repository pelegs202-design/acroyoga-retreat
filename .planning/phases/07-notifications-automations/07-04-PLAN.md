**Create src/app/api/cron/email-drip/route.ts:**
- GET handler with CRON_SECRET Bearer token guard.
- `export const runtime = 'nodejs'`
- Query dripEnrollments: `WHERE channel = 'email' AND next_fire_at <= now() AND completed_at IS NULL AND cancelled_at IS NULL LIMIT 50`.
- For each due enrollment:
  1. Check notificationPreferences for the user (if userId is set). Skip if emailMarketing=false AND this is a marketing sequence (email_nurture). Always send for transactional (email_challenge_reminders).
  2. Look up sequence step from DRIP_SEQUENCES.
  3. Choose the correct React Email template based on sequenceType + currentStep:
     - `email_nurture` → NurtureStep template with archetype from metadata
     - `email_challenge_reminders` → SessionReminder template (compute isEve based on step pattern: even steps = day-before, odd = morning-of)
  4. Generate unsub token for marketing emails.
  5. Call `sendMarketingEmail` or `sendTransactionalEmail` as appropriate.
  6. On success: increment currentStep.
     - For `email_nurture`: If currentStep >= totalSteps (12), reset to step 6 (per locked decision: "Pre-written rotating templates that cycle"). Set nextFireAt to 14 days from now. Do NOT set completedAt — nurture never stops.
     - For `email_challenge_reminders`: Steps are dynamic based on cohort schedule. Compute next session date from metadata.cohortStartDate. Each session gets 2 steps (day-before + morning-of). When all sessions complete, set completedAt. Also send CompletionCertificate email on the final step.
  7. On Resend error: log error, set nextFireAt to 1 hour from now (retry). Do NOT fail the cron.
- Return JSON: `{ processed: N, sent: N, errors: N }`.
- Always return 200.

**Create src/app/api/unsubscribe/route.ts:**
- **GET handler** (one-click from email link): Accepts `?token=xxx&uid=xxx` query params.
  1. Verify token using verifyUnsubToken(token, uid).
  2. If valid: update notificationPreferences set emailMarketing=false for that userId. If no preferences row exists, insert one with emailMarketing=false (other defaults stay ON).
  3. Cancel all active email_nurture drip enrollments for that user's leads.
  4. Return a simple HTML page: "You've been unsubscribed from marketing emails. You will still receive transactional emails (confirmations, reminders)." Bilingual if possible (check Accept-Language header).
  5. If invalid token: return 400 with "Invalid unsubscribe link."
- **POST handler** (List-Unsubscribe-Post one-click): Same logic as GET but returns 200 JSON instead of HTML page. Per RFC 8058, email clients send POST for one-click unsubscribe.
- Per locked decision: "Transactional emails (confirmations, password resets) always send regardless of marketing toggle."
