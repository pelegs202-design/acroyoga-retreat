**Create src/app/api/push/subscribe/route.ts:**
- POST handler: Requires authenticated session (use getAuthSession pattern from existing routes).
- Accepts JSON body: `{ endpoint: string, p256dh: string, auth: string }`.
- Validates all three fields are non-empty strings.
- Upserts into pushSubscriptions table — use `onConflictDoUpdate` on endpoint unique constraint to update p256dh/auth if subscription refreshes.
- Include userAgent from request headers.
- Returns 201 on success.

**Create src/app/api/cron/push-batch/route.ts:**
- GET handler with CRON_SECRET Bearer token guard (per research Pattern 1).
- Set `export const runtime = 'nodejs'` (required for web-push).
- Query pushQueue for unsent items: `WHERE sent_at IS NULL AND (held_until IS NULL OR held_until <= now())` — limit 100 per cron run.
- **Batching logic:** Group queued items by (userId, eventType). If multiple items share same userId + eventType:
  - For 'new_message': collapse into "X new messages" with deepLink to /messages
  - For 'jam_rsvp': collapse into "X jam updates" with deepLink to /jams
  - For other types: send individually
- For each final notification to send:
  1. Look up all pushSubscriptions for that userId
  2. Look up notificationPreferences — skip if pushEnabled=false
  3. Call sendPushToUser for each subscription
  4. Catch 'SUBSCRIPTION_EXPIRED' errors and delete that subscription from DB
  5. Mark all batched pushQueue rows as sent (sentAt = now())
- **Quiet hours re-check:** Even though queuePushNotification sets heldUntil at queue time, the cron should also verify current Israel time vs user's quiet hours — belt and suspenders.
- Return JSON with count of notifications sent.
- Log errors but never throw — cron must return 200 to prevent Vercel retry storms.

**Create src/hooks/usePushPermission.ts:**
- Client-side hook that manages push permission state.
- Exports `usePushPermission()` returning `{ promptForPush: () => Promise<void>, permissionState: 'default' | 'granted' | 'denied' | 'unsupported' }`.
- `promptForPush` checks: (1) browser supports push + service worker, (2) Notification.permission !== 'denied', (3) service worker is ready. If all pass: calls `Notification.requestPermission()`. On 'granted': subscribes via `reg.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: NEXT_PUBLIC_VAPID_PUBLIC_KEY })`, extracts endpoint/p256dh/auth, POSTs to /api/push/subscribe.
- Per locked decision: permission prompt ONLY after first meaningful action (first RSVP or first message sent). The hook itself does NOT auto-prompt — callers invoke `promptForPush()` at the right moment.
- p256dh and auth extraction: use `btoa(String.fromCharCode(...new Uint8Array(subscription.getKey('p256dh')!)))` pattern from research.

**Modify src/app/api/messages/[conversationId]/route.ts** (POST handler — sending a message):
- After successfully inserting a new message into directMessages:
  1. Determine the recipient userId (the other participant in the conversation, not the sender).
  2. Look up the sender's name for the notification body.
  3. Call `queuePushNotification(recipientUserId, 'new_message', 'New message from {senderName}', messageText.substring(0, 100), '/messages/{conversationId}', 'msg_{conversationId}')`.
  4. The batchKey groups messages in the same conversation for batching.
- Import queuePushNotification from @/lib/notifications.
- Per locked decision: tone is minimal — "New message from {name}", no emoji.

**Modify src/app/api/jams/[id]/rsvp/route.ts** (POST handler — RSVP to a jam):
- After successful RSVP (confirmed or waitlist):
  1. Look up the jam host's userId.
  2. Queue push for host: `queuePushNotification(hostUserId, 'jam_rsvp', '{userName} joined your jam', 'New RSVP for {jamDate} at {location}', '/jams/{jamId}', 'jam_{jamId}')`.
- After waitlist promotion (status changes from 'waitlist' to 'confirmed'):
  1. Queue push for the promoted user: `queuePushNotification(promotedUserId, 'jam_rsvp', 'You\'re in!', 'A spot opened up for {jamDate} at {location}', '/jams/{jamId}')`.
- After jam cancellation by host (if that handler exists):
  1. Queue push for all confirmed attendees.
- Per locked decision: language matches user's preferredLocale. Check user.preferredLocale and use Hebrew or English notification text accordingly.

**Modify src/app/api/reviews/route.ts** (POST handler — leaving a review):
- After successfully inserting a review:
  1. Queue push for the reviewee: `queuePushNotification(revieweeId, 'review', 'New feedback received', 'Someone left you feedback', '/profile/{revieweeId}')`.
  2. No name in the notification body — reviews are private per existing design (thumbs up/down only).

**Modify src/app/api/user/update-profile/route.ts** (POST handler — partner-match trigger):
- After successfully updating city, role, or level fields:
  1. Only fire partner-match logic if at least one of city, role, or level was included in the update AND has a non-empty value (skip if user is clearing a field).
  2. Read the user's updated city, role, and level from the database (or from the request body since we just wrote them).
  3. Query for other users who match the updated user's profile complement:
     - Same city (exact match).
     - Complementary role: if updated user is 'base', match 'flyer' and 'both'; if 'flyer', match 'base' and 'both'; if 'both', match all roles.
     - Same or adjacent level: if 'beginner', match 'beginner' and 'intermediate'; if 'intermediate', match all; if 'advanced', match 'intermediate' and 'advanced'.
  4. For each matched user, check the pushQueue table: has this matched user already received a 'partner_match' push today (queued_at >= start of today Israel time)? If yes, skip per locked decision: "Max 1 partner-match push per day, even if multiple join."
  5. For each eligible matched user (not rate-limited), call `queuePushNotification(matchedUserId, 'partner_match', 'New partner near you', 'A new {role} in {city} just joined', '/members/{updatedUserId}')`.
  6. Use the updated user's name and city for the notification body. Deep link to the new member's profile.
- Import queuePushNotification from @/lib/notifications, db and schema from @/lib/db.
- Wrap the entire partner-match block in try/catch — non-blocking. Profile update must succeed even if partner-match notifications fail.
- Per locked decision: "Partner activity matching: City + role + level — precise matching, only notify when new member complements the user."

All push queue operations are non-blocking — wrap in try/catch, log errors, never fail the main API response if push queueing fails. Per decision from 04-02: "Email failure on waitlist promotion is non-blocking" — apply same pattern for push.
