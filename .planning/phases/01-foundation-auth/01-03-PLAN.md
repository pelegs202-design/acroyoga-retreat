    Create `app/api/user/accept-tos/route.ts`:
    - POST handler that validates session via auth.api.getSession({ headers })
    - Updates the user's tosAcceptedAt field to current timestamp
    - Also inserts a row in tosAcceptances table with userId, tosVersion "v1", acceptedAt, and IP address from request headers
    - Returns 200 on success, 401 if no session
    - After updating tosAcceptedAt, the session must be refreshed — return a flag or header so the client knows to refetch session

    Create `app/api/user/update-profile/route.ts`:
    - POST handler that validates session via auth.api.getSession({ headers })
    - Returns 401 if no session
    - Accepts JSON body with optional fields: { city?: string, role?: string, level?: string, preferredLocale?: string }
    - Validate role is one of "base", "flyer", "both" (if provided)
    - Validate level is one of "beginner", "intermediate", "advanced" (if provided)
    - Validate preferredLocale is one of "en", "he" (if provided)
    - Use the Drizzle `db` instance to directly update the user row: `db.update(user).set({ city, role, level, preferredLocale }).where(eq(user.id, session.user.id))`
    - NOTE: We use a custom API route (not `authClient.updateUser()`) because `preferredLocale` has `input: false` in Better Auth config and cannot be updated via the client SDK. Using one route for all onboarding fields keeps the save atomic.
    - Returns 200 with the updated fields on success

    Create `src/components/auth/TosAcceptance.tsx`:
    - "use client" full-screen component per user decision: "Separate full-screen step after signup — not an inline checkbox"
    - Display the combined TOS + Privacy Policy content (use useTranslations for the TOS text or hardcode a reasonable bilingual TOS covering: service description, user conduct in community, data collection, privacy, account termination)
    - Two buttons at bottom: "I Accept" (primary, prominent) and "Decline" (secondary, muted)
    - On Accept: POST to /api/user/accept-tos, then redirect to `/${locale}/onboarding`
    - On Decline: redirect to home page (`/${locale}/`) — per user decision they "can browse public content but cannot create profile, message, or RSVP"
    - Scrollable content area, dark theme

    TOS content to include (bilingual — get from messages/en.json and messages/he.json):
    - Add a new "tosContent" namespace to both message files with sections: intro, communityConduct, dataCollection, privacy, accountTermination, acceptance
    - Per user decision: "Combined TOS + Privacy Policy in one document" and "TOS should cover community conduct basics since the platform replaces WhatsApp groups"

    Create `app/[locale]/(auth)/tos/page.tsx`:
    - Server component that checks session. If no session, redirect to sign-in. If tosAcceptedAt already set, redirect to dashboard.

    - Progress indicator showing current step
    - On complete: POST to `/api/user/update-profile` with `{ city, role, level, preferredLocale: currentLocale }` where currentLocale comes from `useLocale()` (next-intl). This persists the user's language preference alongside their onboarding data.
    - On success response (200): redirect to `/${locale}/dashboard`
    - On error response: display error message, allow retry
    - Include "Skip for now" option at every step per user decision about flexibility. Skip sends POST with only `{ preferredLocale: currentLocale }` (saves locale even if user skips onboarding fields).
    - Per user decision: "Onboarding wizard fields (city, role, level) should align with Phase 3's profile schema so no migration is needed later"
    - Use useTranslations("onboarding") for all text
