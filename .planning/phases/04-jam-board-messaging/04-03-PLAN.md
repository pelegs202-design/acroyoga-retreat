**1. Create `src/lib/dm-channel.ts`** — shared helper for deterministic channel names:
```
export function dmChannel(userA: string, userB: string): string {
  return `dm::${[userA, userB].sort().join('::')}`;
}
```
This MUST be imported by both client and server code to prevent channel name mismatch (Pitfall 2 from research).

**2. Create `src/app/api/ably-token/route.ts`** — GET endpoint:
- Auth required (session check via `auth.api.getSession({ headers: await headers() })`)
- If no session, return 401
- Create Ably Rest client: `new Ably.Rest({ key: process.env.ABLY_API_KEY! })`
- Create token request:
  ```
  const tokenRequest = await rest.auth.createTokenRequest({
    clientId: session.user.id,
    capability: { 'dm::*': ['publish', 'subscribe'] },
    ttl: 3600000, // 1 hour
  });
  ```
- Return `NextResponse.json(tokenRequest)`
- CRITICAL: Never expose ABLY_API_KEY to client. This endpoint is the ONLY way clients get Ably access.

**3. Create `src/components/messages/AblyClientProvider.tsx`** — 'use client' component:
- Import `AblyProvider` from `'ably/react'` and `Ably` from `'ably'`
- Create Ably Realtime client OUTSIDE the component (module scope):
  ```
  const client = new Ably.Realtime({
    authUrl: '/api/ably-token',
    authMethod: 'GET',
    autoConnect: typeof window !== 'undefined',
  });
  ```
- `autoConnect: typeof window !== 'undefined'` prevents SSR WebSocket connections (Pitfall 1)
- Export `AblyClientProvider` wrapping children in `<AblyProvider client={client}>`
- This provider will only wrap the messages route subtree (not the entire app)
  </action>
  <verify>Run `npx tsc --noEmit` — no errors. Verify dmChannel('b', 'a') === dmChannel('a', 'b') — deterministic sorting. Verify AblyClientProvider is marked 'use client'.</verify>

<task type="auto">
  <name>Task 2: Create conversation list and message history APIs</name>
  <files>src/app/api/messages/route.ts, src/app/api/messages/[conversationId]/route.ts</files>
  <action>
**1. Create `src/app/api/messages/route.ts`** — GET conversations list:
- Auth required
- Query conversations where participantA = userId OR participantB = userId
- For each conversation, join with user table to get the OTHER participant's name and image
- For each conversation, get the latest message (subquery or separate query: ORDER BY sentAt DESC LIMIT 1)
- For each conversation, compute unread count: count directMessages WHERE senderId != userId AND sentAt > lastReadAt from conversationReads (left join — if no conversationReads row, count all messages from other user)
- Sort conversations by lastMessageAt DESC (most recent first)
- Return: `{ ok: true, conversations: Array<{ id, otherUser: { id, name, image }, lastMessage: { text, sentAt, senderId } | null, unreadCount: number }> }`

**2. Create `src/app/api/messages/[conversationId]/route.ts`** — GET message history + POST send message:

**GET** — Paginated message history:
- Auth required
- Verify user is a participant in this conversation (check participantA or participantB = userId). If not, return 403.
- Accept query param `before` (ISO timestamp cursor for pagination)
- Query directMessages WHERE conversationId = params.conversationId, ORDER BY sentAt DESC, LIMIT 50
- If `before` param provided: add WHERE sentAt < before
- Join with user table for sender name/image
- **Update conversationReads:** Upsert (insert or update) the user's lastReadAt to `new Date()` for this conversation — marks messages as read on thread open
- Return `{ ok: true, messages: Array<{ id, senderId, senderName, senderImage, text, sentAt }> }`
  Messages returned newest-first; client reverses for display.

**POST** — Send a message:
- Auth required
- Verify user is a participant in this conversation. If not, return 403.
- Parse body: `{ text: string }` — validate text is non-empty, max 2000 chars
- If conversation doesn't exist yet AND the route receives a `recipientId` query param, create the conversation lazily (getOrCreateConversation pattern):
  - Sort the two user IDs: `const [a, b] = [userId, recipientId].sort()`
  - Check if conversation exists with participantA = a AND participantB = b
  - If not, insert new conversation row
  - Use the existing or new conversation ID
- Insert directMessage: `{ id: randomUUID(), conversationId, senderId: userId, text, sentAt: new Date() }`
- Update conversation.lastMessageAt to new Date()
- Update sender's conversationReads.lastReadAt to new Date() (sender has read their own message)
- Return `{ ok: true, message: { id, senderId, text, sentAt } }`
- NOTE: The client-side code will publish to Ably AFTER this API call succeeds — the API does NOT publish to Ably (keeps concerns separate: DB is source of truth, Ably is delivery).
  </action>
  <verify>Run `npx tsc --noEmit` — no errors. Verify GET /api/messages route queries conversations for the authenticated user. Verify POST creates conversation lazily on first message (not eagerly on "Message" button click — anti-pattern from research).</verify>
  <done>GET /api/messages returns conversation list with unread counts. GET /api/messages/[id] returns paginated history and marks as read. POST /api/messages/[id] persists message, creates conversation lazily, updates timestamps.</done>
