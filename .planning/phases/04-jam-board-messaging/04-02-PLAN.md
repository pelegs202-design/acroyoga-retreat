<task type="auto">
  <name>Task 2: Create RSVP endpoint with race-safe capacity check and waitlist promotion</name>
  <files>src/app/api/jams/[id]/rsvp/route.ts</files>
  <action>
Create `src/app/api/jams/[id]/rsvp/route.ts` with a single POST handler that handles both join and cancel actions.

**POST /api/jams/[id]/rsvp** — Join or cancel RSVP:
1. Authenticate. Return 401 if no session.
2. Parse `action` from body: must be `'join'` or `'cancel'`.
3. Extract `id` (jamId) from route params.
4. Verify the jam exists. Return 404 if not found.

**For action = 'join':**
5. Check if user already has an active RSVP (status 'confirmed' or 'waitlist') for this jam. Return 409 "Already RSVPed" if so.
6. **Race-safe capacity check:** Since the DB driver is `neon-http` (stateless HTTP, no persistent connection for `SELECT FOR UPDATE` row locks), use this approach:
   - Count confirmed attendees: `SELECT count(*) FROM jam_attendees WHERE jam_id = ? AND status = 'confirmed'`
   - Compare against jam capacity
   - Insert with status 'confirmed' if under capacity, 'waitlist' if at/over capacity
   - The unique constraint on `(jamId, userId)` prevents the double-insert race condition. If two users hit simultaneously and both pass the count check, one insert succeeds and one hits the unique constraint error — catch the unique violation error and return 409.
   - For the "two users both read count=7, capacity=8" race: at community scale (<100 concurrent users), this window is negligible. The unique constraint is the safety net. If truly needed later, switch to neon websocket driver for real transactions.
7. Insert into `jamAttendees` with randomUUID(), the determined status, and current timestamp as joinedAt.
8. Handle unique constraint violation (PostgreSQL error code '23505'): return 409 "Already RSVPed".
9. Return 200 with `{ status: 'confirmed' | 'waitlist' }`.

**For action = 'cancel':**
10. Find the user's active RSVP (status 'confirmed' or 'waitlist'). Return 404 "No RSVP found" if none.
11. **4-hour cancellation lock:** Check if jam's `scheduledAt` is within 4 hours from now.
    ```typescript
    const CANCEL_LOCK_MS = 4 * 60 * 60 * 1000;
    if (Date.now() > new Date(jam.scheduledAt).getTime() - CANCEL_LOCK_MS) {
      return 403 "Cannot cancel within 4 hours of jam start"
    }
    ```
    Server-side only — never trust client time for this check.
12. Update the user's RSVP status to 'cancelled'.
13. **Waitlist auto-promotion (FIFO):** If the cancelled user was 'confirmed':
    - Query first waitlisted attendee ordered by `joinedAt ASC LIMIT 1`
    - If found, update their status to 'confirmed'
    - Send promotion email via Resend (import resend + FROM_EMAIL from auth.ts pattern):
      ```typescript
      // Look up promoted user's email
      // await the send and log errors (NOT void — per research pitfall #5)
      try {
        await resend.emails.send({
          from: FROM_EMAIL,
          to: promotedUser.email,
          subject: 'A spot opened up for you!',
          text: `Great news — a spot opened in the jam on ${formatDate(jam.scheduledAt)}. You're confirmed!`,
        });
      } catch (err) {
        console.error('[rsvp] Failed to send promotion email:', err);
        // User is already confirmed in DB — email failure is non-blocking
      }
      ```
    - Create a local Resend instance using the same pattern as auth.ts (check process.env.RESEND_API_KEY).
14. Return 200 with `{ status: 'cancelled', promoted: !!nextInLine }`.

Important: Import `Resend` from 'resend' and create the instance locally in this file (same pattern as auth.ts). Use `FROM_EMAIL` env var pattern.
